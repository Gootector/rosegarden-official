How to get involved in Rosegarden development

We're always looking for people willing to participate on the project. If you're interested, here are a few indications on how to start.

First steps

Subscribe to the rosegarden-devel mailing list.

As usual, documentation is too scarce, but there are some design documents in the docs/ directory of the source tree. You can also browse some automatically-generated code reference documentation. We try to keep it up to date, but you can generate it youself with doxygen, the Doxyfile is in CVS.

If you're stuck somewhere or don't understand something, just ask on the mailing-list.

What next ?

Pick something you're interested in doing. You'll have to do that yourself. As strange as it may sound we're not well placed to tell you what to do. First of all, we have no desire to do so, and experience shows that it simply doesn't work that way. People say "Ok, I'll try" and disappear, never to be heard of again :-). So if you're really interested in working with us, it's better if you choose your own job.

If you want to see what we should be currently working on then have a look at our (probably not quite up to date) task lists.

Just remember that contributions of any kind are always welcome. You don't need to ask for permission. If the first thing you send to the list is a patch, we won't blame you for failing to introduce yourself properly, quite the contrary :-).

If you're not too sure about what you intend to do (is it a good idea, are some people already working on this...), then ask on the list.

A bit of discussion before coding never hurts. If what you want to do involves non-trivial design, then it's probably better to ask for comments on the list as well.

But I can't code...

If you're a good writer, you can write some documentation. If you're good at drawing, you can draw icons. If you have experience with other sequencers (Logic, Cubase, Cakewalk), and especially if you're a musician, you can provide us with expertise on the matter, and answer some of our questions on which UI is best.

If you're not any of this, well, it's probably better if you get some programming skills elsewhere and come back. Rosegarden's code is fairly complex, and not a good place to learn C++ or even Qt/KDE programming. For these last two, look at the Qt documentation and KDE Developer's Corner.

A few things to avoid :-)

  • Sending cool ideas ("hey wouldn't it be cool if...") with no apparent willingness to implement them will be frowned upon. It's not that we don't want cool ideas, we like cool ideas, it's more that we need to get Rosegarden to a reasonable level of competency before we can implement them. Have a search through the development mailing list archives to find out what we have talked about over the last few months and what cool ideas we've ignored or are thinking of doing later on. If you think we're missing something (either in the head or in the code) then please don't be backward in coming forward.
  • As a corollary to "pick something you're interested in doing", don't email us asking for permission to help, or offering help, expecting us to guide you or point you to something to do. If you want to help, you'll have to be a bit more autonomous and take initiatives :-).
